Key Steps Explained

Step 4f: Auth Middleware Injection

The platform automatically injects authentication middleware into every deployed app. This is a Vercel Edge Middleware that checks if the visitor has a valid Supabase Auth session with the correct tenant. No company employee published app is ever publicly accessible — this is enforced at the platform level, not by the app code.

File Snapshot

Every deployment stores a snapshot of the source files in the deployments table. This enables: rollback to any version, Claude can get_project() to read and modify existing apps, and the admin dashboard shows full deploy history with diffs.

Update Flow (Existing Projects)

When an employee says "update the sales dashboard", Claude calls get_project() to fetch the current source, makes modifications, then calls deploy() again. The platform detects the existing Vercel project and does an update deployment instead of creating a new one. Same URL, new version.

Agent Deploy (Variant)

For agents, the flow is similar but instead of deploy(), Claude calls schedule_agent(). The platform deploys the agent code as a Vercel Serverless Function + Cron Job (or Supabase Edge Function for webhook triggers). The agent entry in the DB tracks schedule, status, and error history.
