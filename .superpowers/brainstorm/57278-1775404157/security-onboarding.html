Section 6: Security Model & Company Onboarding

How a company gets set up, and how every layer stays secure.

Company Onboarding Flow (Admin runs once)
Step 1: Admin signs up on your platform
   │
   ├──► Creates tenant (company name, slug, domain)
   ├──► Connects auth provider (Google Workspace / Azure AD / SAML)
   │    └── Supabase Auth configured with company's IdP
   ├──► Connects Vercel team account
   │    └── OAuth flow → platform gets deploy token scoped to their team
   │
   ▼
Step 2: Admin configures secrets
   │
   ├──► Adds API keys (OpenAI, Stripe, Slack webhook, etc.)
   │    └── Each encrypted with AES-256, stored in secrets table
   ├──► Sets spending limits (optional)
   │    └── Max deploys/day, max API calls/month per user
   │
   ▼
Step 3: Platform generates config files for Claude
   │
   ├──► CLAUDE.md template (downloaded or auto-configured)
   │    Contains:
   │    • Company conventions & coding standards
   │    • "When publishing, use the deploy tool"
   │    • "Available secrets: OPENAI_API_KEY, STRIPE_KEY, ..."
   │    • "All apps must be in Hebrew RTL" (if set)
   │    • "Database is available via provision_database tool"
   │
   ├──► MCP server config
   │    • For Claude Code: .mcp.json with server URL + auth token
   │    • For Claude.ai: Remote MCP OAuth registration
   │
   ▼
Step 4: Admin invites employees
   │
   ├──► Email invites or auto-join via company domain
   ├──► Each employee gets:
   │    • Access to dashboard (project gallery)
   │    • Their own MCP auth token (scoped to their user + tenant)
   │    • Claude setup instructions (1-click for Claude.ai, CLI command for Claude Code)
   │
   ▼
Step 5: Employees start building ✨

Security Layers

Layer 1: Identity — Who are you?

Dashboard access Supabase Auth via company SSO (Google/Azure/SAML). Only verified @company.com emails. MCP server calls JWT token with tenant_id + user_id + role. Issued at setup, refreshed automatically. Published apps Injected auth middleware checks Supabase session. Redirects to SSO login if not authenticated.

Layer 2: Authorization — What can you do?

Roles admin — manage secrets, users, see all analytics, kill agents.
member — create projects, deploy, see project gallery. RLS enforcement Every DB query filtered by tenant_id from JWT. Cross-tenant access impossible. Secret access Members can reference secrets by name in deploys. Only admins can add/view/rotate actual values.

Layer 3: Data Protection — How is data safe?

Secrets AES-256 encrypted at rest. Decryption key in platform server env only. Never logged, never sent to Claude. Project data Each project's DB schema is isolated. Connection strings scoped to one schema. Audit trail Every deploy, secret access, and agent run logged in usage_events. Who did what, when.

Layer 4: Guardrails — What's prevented?

Rate limits Per-user deploy limit (configurable by admin). Prevents runaway costs. Code scanning Platform scans deployed code for hardcoded secrets before pushing to Vercel. Blocks deploy if found. Resource limits Max DB size per project. Max serverless function execution time. Configurable by admin. Agent kill switch Admin can pause/kill any agent instantly from dashboard. Auto-pause after N consecutive errors.

What the CLAUDE.md Looks Like

# Acme Corp — Claude Configuration
# Auto-generated by the platform. Do not edit manually.

## Company
You are working for Acme Corp. When publishing apps, use the MCP tools available to you.

## Deployment
- To publish any app, use the `deploy` tool. Never ask the user to deploy manually.
- Available frameworks: nextjs, vite, static
- All published apps are automatically secured — do not add your own auth.

## Database
- To give an app persistent storage, use `provision_database` first.
- Use Supabase client libraries for database access in app code.

## Secrets
- Available API keys: OPENAI_API_KEY, STRIPE_SECRET_KEY, SLACK_WEBHOOK_URL
- Reference them by name in the `env_vars` field when deploying. Never hardcode.

## Conventions
- Default language: Hebrew (RTL). Use dir="rtl" on html element.
- Use Tailwind CSS for styling.
- Use shadcn/ui components when building UI.
- Always include error handling and loading states.

## Guardrails
- Max 5 deploys per day per user.
- Do not deploy apps that store PII without explicit user consent.
- Do not create agents that run more frequently than every 5 minutes.
