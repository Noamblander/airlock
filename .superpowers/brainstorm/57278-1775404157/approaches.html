Three Architecture Approaches

Each approach uses the same core idea: CLAUDE.md for context + MCP server for actions. The difference is where things run.

A

Vercel + Supabase (Recommended)

The fast path. Claude already knows both deeply.

Hosting: Vercel — static, serverless functions, edge runtime
Database: Supabase — PostgreSQL + Auth + Storage + Realtime
Auth gateway: Supabase Auth (Google, Microsoft, SAML)
MCP Server: Single Next.js app — serves both MCP endpoints + admin dashboard
Agents: Vercel Cron + Supabase Edge Functions

Pros

  • Claude generates Vercel/Supabase code natively — highest quality output
  • Fastest to build (2-4 weeks MVP)
  • Generous free tiers for starting
  • Single codebase: MCP server + dashboard
  • Supabase RLS = row-level security per tenant built-in

Cons

  • Vendor lock-in to Vercel + Supabase
  • Costs grow with scale (but fine for 10-50 users)
  • Cron-based agents limited to 1-min intervals
B

Cloudflare Ecosystem

The scale path. Edge-first, built-in auth gateway, best for agents.

Hosting: Cloudflare Pages + Workers
Database: D1 (SQL) + R2 (storage) + KV (config)
Auth gateway: Cloudflare Access — sits in front of ALL published URLs
MCP Server: Cloudflare Worker with Durable Objects
Agents: Durable Objects — persistent, stateful, real-time

Pros

  • Cloudflare Access solves secure-URL auth elegantly (SSO gateway)
  • Durable Objects = perfect agent runtime (persistent, event-driven)
  • Cheapest at scale by far
  • Global edge = fast everywhere

Cons

  • D1 is less mature than PostgreSQL
  • Claude generates less reliable Cloudflare code than Vercel/Supabase
  • Steeper learning curve
  • Longer to build (4-6 weeks MVP)
C

Hybrid: Vercel + Supabase + Cloudflare Access

The balanced path. Best DX from A, best auth from B.

Hosting: Vercel (behind Cloudflare proxy)
Database: Supabase
Auth gateway: Cloudflare Access in front of all published apps
MCP Server: Next.js on Vercel
Agents: Vercel Cron + Supabase Edge Functions

Pros

  • Cloudflare Access = zero-trust auth on every published URL
  • Vercel/Supabase DX for building
  • Easy to migrate agent runtime to Cloudflare later

Cons

  • More moving parts (3 vendors instead of 2)
  • DNS/proxy config adds complexity
  • Still has Vercel's agent limitations
